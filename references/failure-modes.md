# Common Failure Modes

| Symptom | Likely cause | Fix |
|---|---|---|
| **Concept art quality (Step 2)** | | |
| Concept came back as detailed AI illustration with smooth gradients, fine fur strands, sub-pixel detail | The `Pixel art rendering:` line was missing or weak in the imagegen prompt. | Re-run with the line included verbatim from `concept-prompt-template.md`. It must (a) name positive references like Pokemon Crystal / Stardew Valley / Earthbound NES, (b) forbid anti-aliasing / sub-pixel detail / gradient shading in the same paragraph. |
| Output doesn't match a specific game style the user requested (e.g., "River City Ransom") | The game name wasn't passed verbatim into the Subject line, OR it was buried under abstract structural parameters (sprite size / palette count / shading description) that dilute the reference. | Re-roll with the game name(s) verbatim in the Subject line plus character-specific descriptors that match that game (e.g., for Kunio: "spiky black hair, white tank top, denim jeans, red sneakers"). For best fidelity, ask the user to attach a reference image (Variation 3). |
| Multiple subjects appeared (companion pet, second character, crowd) | The single-subject constraint wasn't emphasized strongly enough. | Re-roll with "ONE subject only, NO additional characters, NO companions, NO crowd" in Constraints. The image model defaults to scenes; single-subject must be forced explicitly. |
| Concept's head or feet clipped at frame edges | 15% margin wasn't enforced for tall features (hair, hats, ribbons, antennae) or feet. | Re-roll with stricter Composition language. |
| Reference image's identity wasn't preserved (Variation 3) | The Input images line didn't specify identity preservation. | Re-roll with explicit "Image 1: preserve subject's identity, silhouette, costume, key features. Transform ONLY the rendering style to pixel art." |
| Bg isn't a flat solid chroma-key color (came back with scenery, gradient) | Model didn't honor `Scene/backdrop` chroma-key request. | Re-roll. The prompt template includes the documented chroma-key bg block; verify it was included verbatim. |
| **Background removal (Step 4 — after user accepts)** | | |
| Holes in character (patches transparent — e.g., white shirt highlights gone, sash punched out) | Used a simple color-distance bg-removal with tolerance > 0. Tolerance window included character pixels close to the bg color. | Always use the imagegen skill's helper at `$CODEX_HOME/skills/.system/imagegen/scripts/remove_chroma_key.py` with `--soft-matte --transparent-threshold 12 --opaque-threshold 220 --despill`. Soft matte preserves character pixels via partial alpha. **Never write a custom color-distance matcher.** |
| Thin colored fringe around the silhouette | Despill wasn't strong enough OR the matte's transparent-threshold was too low | Retry with `--edge-contract 1`. Use `--edge-feather 0.25` only if edge is visibly stair-stepped AND subject isn't shiny/reflective. |
| **Pixel snapper (Step 5)** | | |
| Snapped colors are muddy / shadows merged with body colors | The default `--colors 16` k-means cap may be too tight for color-rich subjects. | Raise `--colors` to 24-32 for more distinct shading bands. |
| Snapped output is noisy / grainy / blocks-don't-cluster | Should be fixed by the current k-means-then-mode algorithm. If still occurring, your `snap_pixels.py` may be an older version that pre-quantized via FASTOCTREE or used uniform-NxN cuts. | Update to the current version: k-means quantize full-res first, then grid-aligned `walk` for cuts, then mode-per-cell. |
| Auto-detected pixel size is wrong (output too coarse or too fine) | The gradient-profile detection picked the wrong step size. | Override with `--pixel-size N` based on inspection of the concept's actual native pixel granularity. |
| Snapped output is SMALL (e.g., 100x100) — looks "low resolution" | This is intentional — Sprite Fusion native logical resolution. Every pixel in the file IS one logical pixel. | Use `upscale.py` to NEAREST-upscale for display. The small file is the canonical game-engine asset. |
| **Upscale (Step 6)** | | |
| Upscaled output is distorted (portrait squashed into square or vice versa) | Old `upscale.py` stretched the input into the target dimensions ignoring aspect ratio. | Current `upscale.py` does canvas-fit: largest integer scale that fits the input inside the canvas, centered with transparent padding. Update to the current version. |
| Upscaled output has uneven block widths | NEAREST upscale ratio isn't an integer (e.g., 1024 / 114 = 8.98). | Expected for arbitrary target sizes. The per-pixel art is still clean; only block widths vary by ±1. For perfectly uniform blocks, use `--scale N` for an integer scale factor (output = `input × N`, no canvas). |
| **Agent behavior** | | |
| Agent generated art using `PIL.ImageDraw` instead of `imagegen` | Agent didn't read the SKILL's `imagegen` requirement | Stop. Use `image_gen` for the concept. The Python scripts are for post-processing only. |
| Agent skipped Step 3 user review and ran bg removal + snap + upscale automatically | Workflow step missed | The user MUST review the RAW concept (with chroma-key bg) and accept BEFORE Step 4 runs. Don't auto-proceed. |
| Agent reused PNGs from a prior project | Pattern-matching on existing assets | Every asset generates fresh via `imagegen`. Delete reused assets, regenerate. |
| Agent wrote a custom bg-removal helper | Agent saw the pipeline and assumed it needed its own tool | The imagegen skill's `remove_chroma_key.py` is the bg-removal tool. Never duplicate. Path: `$CODEX_HOME/skills/.system/imagegen/scripts/remove_chroma_key.py`. |
| Agent produced a generic / boring concept for a vague prompt | Step 1 enrichment wasn't thorough enough | The agent's prompt-enrichment is the creative layer. For a vague prompt, add visual specificity (color palette, distinctive features, pose) using world knowledge. See translation examples in `concept-prompt-template.md`. |
